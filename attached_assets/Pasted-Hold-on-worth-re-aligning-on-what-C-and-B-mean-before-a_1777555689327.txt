## Option C — Location cruft strip (per-uni text-cleaning override for ACAP)

**What's broken:** 4 of the 11 staged ACAP courses currently render their location with a literal annotation suffix. From the most recent staged data:

```
Master of Social Work (Qualifying)            Adelaide, Melbourne, Sydney, Perth^ ^Available in Perth from Trimester 3, 2026
Graduate Diploma of Professional Accounting   Perth^ ^Available in Perth from Trimester 3, 2026
Graduate Certificate of Professional Accounting   Perth^ ^Available in Perth from Trimester 3, 2026
Graduate Certificate of Human Services        Adelaide, Melbourne, Sydney, Perth^ ^Available in Perth from Trimester 3, 2026
```

The `^ ^Available in Perth from Trimester 3, 2026` is a footnote annotation on ACAP's website (the `^` is rendered as a superscript footnote marker in CSS, but the static HTML extractor reads it as literal text). It's been in every ACAP scrape log for the past three weeks but only became visible in user-facing data after Phase 4 (when ACAP started staging non-zero courses).

**Why this is C:**
- Per-uni only — ACAP is the only uni with this specific footnote pattern
- No shared-code change — the fix is a YAML override that adds a regex strip pattern to ACAP's text-cleaning config
- No regression sweep needed — by definition only ACAP is affected
- Validates a third per-uni override shape (text-cleaning regex) after Phase 2 (boolean filter) and Phase 5 (behaviour toggle)
- Half-day of work

**What the fix looks like:**

In `acap_41.yaml`:
```yaml
extraction:
  text_cleaning:
    location:
      strip_patterns:
        - '\^\s*\^.*$'    # ACAP footnote: "^ ^Available in Perth from Trimester 3, 2026"
```

In `defaults.yaml`:
```yaml
extraction:
  text_cleaning:
    location:
      strip_patterns: []    # default: no stripping
```

In shared code (the location text post-processor): apply each `strip_patterns` regex to the extracted location text before it gets staged. Read from `uc.extraction.text_cleaning.location.strip_patterns` via the contextvar.

**Validation:**
1. Capture fresh post-Phase-5 baseline
2. Add the YAML override + the strip code in shared code
3. Run regression sweep with `--expected-slugs acap`. Expected: 22 clean, 1 acknowledged-dirty (ACAP location values for those 4 courses change from `Perth^ ^...` to `Perth`)
4. Spot-check the 4 ACAP courses in the review UI
5. Merge

## Option B — Duration regex shared-code fix

**What's broken:** the structural duration regex misclassifies values across multiple unis. Specific known instances from the bug reports:

- KBS Graduate Certificate in Accounting → page text says "8 months / 4 subjects / 2 trimesters", regex extracted as `8 Year` (the `[NULLIFIED]` sanity check then cleared it, leaving duration blank)
- KBS Master of Information Technology → page text says "1 year, 8 months / 12 subjects / 5 trimesters", regex extracted as `20 Year` (also nullified)
- ACU Doctor of Philosophy → 12-year duration value extracted from "4 years"
- AIT ICT50220 → 15-year duration value extracted
- ACAP Graduate Diploma of Business Administration → currently displays `16 Month` (the page says "1 year, 4 months")
- ACAP Graduate Diploma of Legal Practice → currently displays `3 Month` (the page says "14 Weeks Full Time")

**Common root cause:** the duration regex grabs numbers without their units, or grabs the wrong number from "X years / Y subjects / Z trimesters" patterns. The regex needs to:
- Always capture both number AND unit together (no orphan numbers defaulting to years)
- Prefer `\d+\s+(year|years|month|months|week|weeks)` matches over `\d+\s+(subject|trimester|semester)` matches
- Split on `/` when the format is `X years / Y subjects` and only take the first segment for years

**Why this is B (not C):**
- Shared code — the regex lives in the structural duration extractor used by every uni
- Multi-uni blast radius — KBS, ACU, AIT, ACAP all show the bug. Other unis without the bug also run through this code path, so the fix could accidentally regress them.
- Requires the regression sweep with `--expected-slugs kbs,acu,ait,acap` (or however many unis turn out to actually be affected when you query the data)
- This is the **first multi-uni expected-dirty sweep test** — every previous migration had ≤1 expected-dirty uni

**Suggested validation order for B:**
1. Capture fresh post-Phase-5 baseline
2. Write a unit test that demonstrates the bug in isolation (input: "8 months / 4 subjects / 2 trimesters" → expected: 8 months, current actual: 8 years)
3. Query the staged data across all 22 baselined unis to enumerate every course with `duration_value >= 5 AND duration_unit = 'years' AND degree_level IN ('Graduate Certificate', 'Graduate Diploma')`. That returns the full list of bug-affected courses across the fleet — don't guess at which unis are affected, query and find out.
4. Fix the regex
5. Verify the unit test passes
6. Run regression sweep with `--expected-slugs <full enumerated list from step 3>`
7. Diff should show ONLY duration_value/duration_unit changes on the enumerated courses, no other field changes anywhere, no other unis dirty
8. If sweep matches expectation: merge
9. If sweep shows duration changes on a uni you didn't expect: investigate — that uni had the bug and you didn't know. Add to enumerated list, re-run.
10. If sweep shows non-duration changes anywhere: real regression, debug before merge
