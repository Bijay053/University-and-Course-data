ielts_overall = 6.5, source: per_course_vision (NOT per_course_vision_cached)

This is a generic "How to apply" marketing image used across the entire site — definitely not specific to the Accounting course. The source is per_course_vision, not per_course_vision_cached, which means the fix you made addressed the cache contamination, but the underlying problem is broader: vision is mining IELTS values from any image on the page, including generic site-wide marketing imagery.
The composite cache key fix prevents one course's cached value from leaking to another. It does not prevent vision from extracting IELTS from the wrong image in the first place. The "how-to-apply" image likely contains text like "IELTS 6.5 / PTE 62 / IELTS bands 6.0" because it's the generic application requirements graphic — and vision is happy to scrape those numbers regardless of what course page it's on.
In this Accounting case the value happens to match what regex also found (6.5), so it's not actively wrong. But the evidence is junk. And on courses where a generic marketing image happens to show different numbers than the actual course requirement, this would silently produce wrong values.
3. What this means
Last night's "0 promoted vision_cached" audit was correct for the question it asked. But it asked the wrong question. The real question is: how many promoted courses got IELTS from per_course_vision (without the _cached suffix) where the image URL doesn't belong to that course?
Updated audit:
sqlSELECT COUNT(*) AS promoted_vision_unrelated_image
FROM courses c
JOIN scraped_courses sc ON sc.course_id = c.id
WHERE sc.extraction_method->>'ielts_overall' IN ('per_course_vision', 'per_course_vision_cached')
  AND sc.status = 'promoted';
If that number is greater than 0, the live catalog has the same issue. The fix is bigger than caching — vision needs a relevance check (does this image's URL path or alt-text relate to this course?) before its IELTS values are accepted.
My honest take: the Bug 1 fix from last night was real progress (cache no longer leaks across courses), but the underlying vision-OCR-mining-generic-images problem wasn't actually solved. We addressed a symptom, not the disease. Bug 2 might be a frontend/caching issue rather than a DB issue — easy to verify with the SQL above.
