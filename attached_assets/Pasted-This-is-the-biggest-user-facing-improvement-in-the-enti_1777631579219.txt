**Two things flagged in the summary that need addressing before this is fully done:**

**Issue 1 — GitHub push conflict.**

From the summary: "local is 10+ commits ahead of github/main — the bulk_approve endpoint and pipeline fixes from Replit haven't been pushed to GitHub yet, so prod is running from the heredoc script only."

This is the same divergence pattern that caused the bundled-deploy disaster three weeks ago. Prod has code that GitHub doesn't have. If prod ever rebuilds from GitHub (server replacement, container redeploy, anything), all of this work disappears.

What needs to happen:

```
cd ~/University-and-Course-data
git status                                # confirm what's local-only
git log origin/main..HEAD --oneline       # see the divergent commits

# Either fast-forward push if local is a clean superset:
git push origin main

# Or if there's been remote work (unlikely but check):
git fetch origin
git log HEAD..origin/main --oneline       # any new remote commits?
```

If it's a clean superset, just push. If there's remote-only commits we need to reconcile carefully — don't force-push without knowing what we'd lose. The bulk_approve script and pipeline fixes need to land on GitHub today.

**Issue 2 — The pending/review queue is not actually empty.**

The live count summary showed `legacy_gap = 0` and `pending_ready = 0`, which is true for the categories we ran bulk_approve on. But the summary also flagged:

> 36 VIT pending/review fresh-scrape rows need admin review before promotion

That's 55 courses sitting in `auto_publish_status = 'needs_review'` that we didn't bulk-approve because they have unknown quality issues. Worth running the diagnostic:

```
sudo -u postgres psql university_portal -c "
SELECT u.name AS uni,
       sc.id AS sc_id,
       sc.course_name,
       sc.confidence,
       sc.auto_publish_status,
       CASE WHEN sc.international_fee IS NULL THEN 'fee_missing' ELSE NULL END AS fee_issue,
       CASE WHEN sc.ielts_overall IS NULL THEN 'ielts_missing' ELSE NULL END AS ielts_issue,
       CASE WHEN sc.duration IS NULL OR sc.duration = 0 THEN 'duration_missing' ELSE NULL END AS duration_issue
FROM scraped_courses sc
JOIN universities u ON u.id = sc.university_id
WHERE sc.university_id IN (4, 6)
  AND sc.auto_publish_status = 'needs_review'
ORDER BY u.id, sc.confidence;"
```

This reveals what's blocking those 55. If it's all fee-missing on VIT (which would track with the CRICOS-suffix scraper issue we acknowledged-without-investigating earlier), we have a path. If it's mixed issues, each row needs different treatment.
