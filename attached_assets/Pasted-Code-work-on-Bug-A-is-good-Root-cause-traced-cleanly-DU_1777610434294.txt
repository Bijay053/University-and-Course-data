**Issue 2: `git pull origin main` fails with divergent branches.**

```

```

This is the GitHub force-push decision biting us. Local has 8 sequenced commits, origin has 8 bundled commits, divergent histories. `git pull` doesn't know which to apply.

Three options for getting Bug A onto prod:

*Option 1 — Force-push the clean local history to origin first.* Then `git pull origin main` works as fast-forward. This is the cleanest long-term but the most destructive (overwrites origin's bundled-deploy history, though the `archive/bundled-deploy-attempt` tag preserves it).

*Option 2 — Cherry-pick Bug A directly onto local main, push as a separate branch, merge.* Avoids the force-push but creates more divergence on origin.

*Option 3 — Develop Bug A on the dev environment, push to a branch on origin (e.g., `bug-a-duration-slash`), then on prod do `git fetch origin bug-a-duration-slash && git cherry-pick origin/bug-a-duration-slash`.* Cleanest if Bug A is a single commit, doesn't need force-push, and keeps both histories independent.

My recommendation: **Option 1.** Force-push origin to match local. The bundled-deploy history is an embarrassment, not a reference — the tag preserves it for forensic purposes if anyone ever needs it. Clean history going forward is more valuable than preserving the divergence.
