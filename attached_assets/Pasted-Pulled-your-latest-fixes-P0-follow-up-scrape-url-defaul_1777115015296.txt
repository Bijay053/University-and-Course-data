**Asking for: Bug 2 design proposal first, then code.**

From the original ticket I asked for a design doc before coding. That's still the right approach — Bug 2 isn't a quick patch. Specifically I want to see your thinking on:

1. **Discovery rules.** How does the scraper identify which links on a course page are central admissions pages vs. just outbound nav? URL patterns (`/admissions/fees`, `/fees`, `/entry-requirements`, `/english-requirements`)? Anchor text matching? Both?

2. **Cache key shape.** A central fees page might list every program's fee in one table indexed by course code (KBS uses CRICOS code 110274C etc.). Or a single "international students pay $X" generic statement. Or a fee table per faculty. The cache lookup needs to handle all three. What's the data model?

3. **Per-uni vs. global cache.** Cache lives for one scrape run? One uni forever? Refresh how often? My instinct: per-uni, refreshed per scrape run, lives in memory only.

4. **Pipeline insertion point.** Where does the lookup slot in? My instinct: between the per-course fetch and the staging gate — same place the sibling cache backfill already runs.

5. **Fallback behavior.** If central page exists but doesn't have this specific course's fee, what happens? Reject as no-fee, or stage with central "international students pay X" generic value, or surface for human review?

6. **Scope estimate.** How many days of work is this? If it's >3 days I want to scope it down. If <1 day I want it shipped this week.

Send the design proposal as a markdown doc in your repo or paste it as a Replit message. I'll review and we can converge before you write code. **Don't start coding until we agree on the design.**
