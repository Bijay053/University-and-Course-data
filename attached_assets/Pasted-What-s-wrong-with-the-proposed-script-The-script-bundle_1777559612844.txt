On Fix 2 specifically:
The schema migration script proposed (drop materialised view, alter column, recreate view, grant, update alembic_version) is doing destructive operations on the live production DB with no rollback plan and no sweep validation. The script doesn't even take a backup before the ALTER TABLE. If ROUND(duration::numeric, 2) produces unexpected values on edge cases (NULL handling, infinity, NaN from old float storage), every duration row of every uni is corrupted in the same instant.
The right path for Fix 2:

Clone prod DB to staging (pg_dump → restore to staging instance)
Apply the migration on staging
Capture baseline snapshot from staging
Diff against current prod baseline
Expect: zero behavioural diffs (same values, just stored as Numeric instead of float)
If clean: take prod backup, apply migration to prod with same script, verify, restart services
If diffs: investigate which uni's data drifted before going near prod

This is 30 minutes of work to do safely. The "paste-and-run" version skips all of it and runs destructive DDL on production data straight from a one-liner script.
