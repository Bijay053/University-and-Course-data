**On the workflow drift:**

The English fix and fee fix were shipped together as one deploy. We agreed each shared-code change should ship as its own commit with its own sweep. Bundling two shared-code changes — especially when one of them adds a new code path (`_parse_column_keyed_english_table`) — is a smaller version of the bundled-deploy disaster.

With both changes deployed and the sweep failing, we now can't tell which change caused which unexpected diff. Was it the fee sub-page follower that found new pages on vit? Or the column-keyed English parser unlocking previously-blocked extractions? Both? Different things on each uni?

The answer matters for the fix design. If we have to revert and redo, we don't know which of the two commits to revert.

**Recommendation for going forward:** the next migration ships ONE shared-code change at a time. No bundling, even when changes look related. The discipline isn't about how many commits feel related — it's about being able to attribute every diff in a sweep to a specific change.
