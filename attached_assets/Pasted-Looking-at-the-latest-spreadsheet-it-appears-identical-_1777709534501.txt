**Independent of the Flinders re-scrape, the prod audit SQL is worth running now**

The audit query against `scraped_field_evidence` for `sibling_cache:%` provenance on approved/published English fields doesn't depend on any new scrape — it tells us whether the live 792 catalog already has fabricated English data sitting in production across the 12 unis. That's a handover question we need answered regardless of what Flinders staging does next.

```sql
-- Run on PROD DB
SELECT sc.university_id, u.name,
       sfe.field_key,
       COUNT(*) as cnt
FROM scraped_field_evidence sfe
JOIN scraped_courses sc ON sfe.scraped_course_id = sc.id
JOIN universities u ON sc.university_id = u.id
WHERE sfe.extraction_method LIKE 'sibling_cache:%'
  AND sfe.selected = true
  AND sfe.field_key IN (
      'ielts_overall','ielts_listening','ielts_reading',
      'ielts_writing','ielts_speaking'
  )
  AND sc.status IN ('approved','published')
GROUP BY sc.university_id, u.name, sfe.field_key
ORDER BY sc.university_id, sfe.field_key;
```

- Zero rows: live catalog is clean, only newly-staged rows were ever affected
- Small count (<50 across catalog): targeted manual remediation, doesn't block handover
- Large count (hundreds): coordinated remediation re-scrape across affected unis required, may shift handover timing
