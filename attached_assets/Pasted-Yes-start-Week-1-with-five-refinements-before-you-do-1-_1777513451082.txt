Yes — start Week 1, with five refinements before you do.
1. Don't pause prod fixes during the migration. ACAP is currently broken on the re NameError, KBS has zero fees, JCU stages zero. If Week 1 freezes other work for 5 days, those stay broken. Run the migration in parallel with continued prod fixes — but every prod fix from now until Week 4 lands gets retroactively moved into the per-uni config it should have been in, not left in shared code. Otherwise you're widening the gap you're trying to close.
2. Capture the known-good baseline before touching anything. Before you create the stub configs, do one full scrape run of all 17 bug-reported unis and snapshot the output (course count, every field for every course). That snapshot becomes the regression floor for Weeks 2–4. Without it, Week 2 ("move overrides, verify identical output") has nothing to compare against and the verification step is just "looks fine to me." You only get one chance to capture this baseline cleanly — once Week 2 starts, you can't tell whether a difference is from the migration or from the underlying site changing.
3. Your Tier 3 scoping is right and worth being explicit in code. Discovery settings only — block_url_patterns, allow_url_patterns, fallback_subdomains, sitemap behaviour. Not extraction settings, not text cleaning, not filters, not vision OCR toggles. Running ACAP's trust_vision_ocr: false against a new uni that has perfectly good IELTS images would actively hurt. Suggest splitting the per-uni config schema into two top-level sections — discovery: (replayable across unis in Tier 3) and extraction: (uni-specific, never replayed) — so the boundary is enforced by structure, not discipline.
4. Pick the context-passing pattern upfront, don't mix. Either every extractor takes a uni_config: UniConfig parameter (explicit, verbose, refactor-heavy but easy to reason about), or you use a contextvar / thread-local that gets set at the worker entry point (less plumbing, but every extractor now has hidden global state, and tests need fixtures). Both work. The danger is doing one for some extractors and the other for others — you end up with bugs where a unit test passes a config but production reads from contextvar and gets the wrong one. Pick the explicit-parameter version unless there's a specific reason not to; it's slower to write but you'll spend less time debugging weird "wrong config applied" issues later.
5. Add a shadow-mode step to Week 2 before cutover. When you move an override out of shared code into a per-uni YAML, don't immediately cut over. Run both paths (old shared code + new config-driven code) in parallel for the affected uni and assert the outputs are byte-identical. Only when 5 consecutive runs match, retire the shared-code branch. This catches subtle differences (regex flag mismatches, default-value drift, dict iteration order) that "looks fine to me" verification will miss. Yes, it doubles the runtime during migration; no, that's not a real cost when set against the regression risk.
On your specific question about defaults.yaml staying conservative — yes, that's exactly right and worth making operationally enforced rather than just stated. Suggest: any PR that modifies defaults.yaml requires the regression sweep to run and a human approval of the diff before merge. Any PR that only adds/modifies a file under unis/*.yaml skips the sweep and only re-runs that one uni. The asymmetric cost will naturally push fixes toward per-uni overrides, which is the behaviour you want — promotion to defaults.yaml only happens once an override has been proven on 3+ unis and someone explicitly decides it's the better default.
One last thing worth noting: this work doesn't fix any current bug. It makes the next round of bug fixes safer. If the team or stakeholders are measuring progress in "bugs closed this week," Weeks 1–4 will look like a regression in throughput. Worth flagging that to whoever's tracking it before you start, so the metric doesn't get used against the migration.
