Two possible interpretations:
1. ACAP genuinely changed the images on their site (the cdn-ileicgm.nitrocdn.com URLs look the same as before, so probably not this)
2. The vision OCR pass is non-deterministic across runs (Gemini reading the same image returns different numbers on different days) — this is more likely

If interpretation 2 is correct, that's important for shadow mode. Two scrape runs of the same uni against the same images may produce different vision OCR results. The shadow-mode "5 consecutive runs match byte-identical" criterion needs to either:
- Cache the vision OCR results so both shadow paths see the same image → same result
- Exclude vision-OCR-derived fields from the byte-identical comparison (but compare them with a tolerance)
- Treat a vision-OCR-only difference between runs as expected noise rather than a regression

Worth knowing this BEFORE shadow mode goes live, because otherwise the first ACAP shadow run will produce a diff (vision OCR returned different numbers), the streak counter resets, and you'll spend a day debugging "why is the new code path producing different IELTS values" when the actual answer is "vision OCR is non-deterministic."

Recommend: cache Gemini vision responses on (image_url, model_version) for the duration of a shadow-mode session, so both old and new code paths get the identical OCR result. That isolates the test to "is the new code path equivalent given the same inputs" rather than mixing in OCR drift.
