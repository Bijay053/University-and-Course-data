**The pushback:**

You wrote: "The 16 method-only changes (`extraction_method.duration_term: 'gemini_primary' → None`) are the Bug A.2 fix doing its job — Gemini can no longer quietly override `duration_term` when `duration` is already owned by a structural method."

We established yesterday that `extraction_method.duration_term: 'gemini_primary' → None` is a *provenance metadata* change — not a value change. The actual `duration_term` field on those 16 courses is unchanged. The sweep diff format only labels something method-only when `_VALUE_FIELDS` are unaffected.

So "Gemini can no longer quietly override `duration_term`" is the wrong reading. Gemini was never *overriding* the term value — it was *being credited as the source* in the provenance dict, while the actual term value came from regex/structural extraction. After Bug A.2, Gemini correctly isn't credited as the source anymore (because the orchestrator skips Gemini for those fields when regex won), but the values were always coming from regex, both before and after.

The Bug A.2 fix is real and shipped — the proof is the 8 ADDED courses, not the provenance dict cleanup. The provenance cleanup is a side effect, not the primary effect.

This matters because if we don't get the interpretation right, the next migration's sweep diff might be misread the same way and we'll miss a real signal hiding in metadata churn.
