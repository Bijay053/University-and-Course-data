Bug 2 — Bachelor of Nursing has IELTS=7.0 overall but the band scores are wrong
URL: https://www.torrens.edu.au/courses/health/bachelor-of-nursing
Dashboard shows IELTS 7.0 overall (correct) and PTE 66 / TOEFL 94 (correct on the overalls). But the individual IELTS bands the dashboard surfaces don't match what's actually on the page.
Page wording:

Academic IELTS 7.0 in three components (listening, reading, and speaking) and minimum score of 6.5 in writing; OR OET B in each of the three components (listening, reading, speaking) and C+ minimum in writing OR TOEFL iBT minimum score of 94 OR PTE Academic minimum score of 66

The correct extraction is:

ielts_overall = 7.0
ielts_listening = 7.0
ielts_reading = 7.0
ielts_speaking = 7.0
ielts_writing = 6.5

This is a different per-band pattern than the floor clause Patch 1c was built for. Patch 1c handles "minimum overall band of X with no individual band below Y" — i.e., a single floor that applies to all four bands. Bachelor of Nursing uses a split per-band requirement: three bands at one value, one band (usually writing) at a lower value. The af2ae03 fix doesn't cover this shape.
This phrasing is NOT rare in AU healthcare/nursing programs — it's the standard Nursing & Midwifery Board of Australia (NMBA) / AHPRA wording, used on most Bachelor of Nursing pages across the country. So this isn't a Torrens-specific issue; it'll show up on UTS, ACU, La Trobe, USQ, JCU, etc., wherever Nursing is offered.
Suggested fix — extend Patch 1c with a "split-component" pattern. Approximate regex sketch (verify against 5 real examples first as you did with af2ae03):
python# Match "X in [three|N] components (listening, reading, speaking)
#        and (minimum|min|at least) score of Y in writing"
# Also handle the inverse: "Y in writing and X in listening/reading/speaking"
_SPLIT_COMPONENT_IELTS_RE = re.compile(r"""
    ielts(?:\s+academic)?
    .*?
    (?P<higher>\d+\.?\d*)
    \s+in\s+(?:three|3|all)?\s*components?\s*
    \(?\s*(?P<bands>(?:listening|reading|speaking|writing)
                  (?:\s*,\s*(?:and\s+)?(?:listening|reading|speaking|writing))+)\s*\)?
    \s+and\s+(?:minimum\s+score\s+of\s+|min(?:imum)?\s+|at\s+least\s+)?
    (?P<lower>\d+\.?\d*)
    \s+in\s+(?P<remaining_band>listening|reading|speaking|writing)
""", re.IGNORECASE | re.VERBOSE | re.DOTALL)
Logic:

Identify the higher value and which 3 bands it applies to (parse the bands group)
Identify the lower value and which 1 band it applies to (remaining_band)
Set those four band values explicitly
ielts_overall should already be set by Pattern 1/1b/1c on the same page; if it's null, fall back to the higher value

Same caveat as af2ae03: per-band values need to land in the normalized dict of the ielts_overall ExtractionResult so they don't create separate evidence rows that the sibling cache could propagate. Source-type gating still pending.
Test cases for the new pattern (run before deploying):

Don't ship without the fourth one passing — the new pattern needs to coexist with the floor pattern, not replace it.

The bigger thing this points to
The Bachelor of Nursing case isn't a one-off. AU Nursing programs across the fleet use this exact NMBA phrasing. Worth a query against the staged data before Bug 2 ships:
sqlSELECT university_id, course_name, ielts_overall, 
       ielts_listening, ielts_reading, ielts_writing, ielts_speaking
FROM scraped_courses
WHERE course_name ILIKE '%nursing%'
  AND status IN ('pending', 'approved')
ORDER BY university_id, course_name;
If multiple unis show ielts_overall populated but bands mismatched against the page text (you can spot-check 3-4 of them by hand), that's the blast radius. The fix is a shared-code regex addition, so the regression sweep across all 23 baselined unis is the right gate. Expect dirty on every uni with a Nursing program after the fix lands.
