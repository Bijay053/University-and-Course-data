## What's happening

First CSU run today (commit `edc05e20`, concurrency=5):

Second CSU run today (commit `c8f542a`, concurrency=10):

Log evidence:

The text is the un-hydrated React shell — CSU's React app never fired its data fetch. Every course gets `ielts=6.0` from central English page and nothing else — no fees, no intakes, no duration, no campus locations.

## Why concurrency=10 made it worse

Hypothesis: CSU has bot detection or rate limiting. Previous successful run was the 1st CSU attempt of the day with low concurrency. After multiple scrape attempts (3+ today), CSU likely throttled our IP. The 10 concurrent connections hitting it simultaneously may have triggered harder blocks.

## What's needed

Not another timeout/concurrency tweak. CSU needs a different scraping strategy:

**Option A: API-based scrape**
- CSU's React app fetches data from `https://study.csu.edu.au/api/...` endpoints 
- Inspect network tab in browser devtools on a CSU course page
- Find the JSON API endpoint that feeds the React app
- Bypass the browser entirely — fetch JSON directly from the API
- Massively faster, no Cloudflare interaction, no React hydration wait

**Option B: Wait selector instead of networkidle**
- networkidle is failing because CSU's React app keeps making background requests
- Use `page.wait_for_selector('.course-detail-loaded', timeout=30000)` or whatever the actual content marker is
- Once that selector appears, the data is rendered
- Much faster than waiting for ALL network to be idle

**Option C: Reduce concurrency for CSU specifically**
- If CSU has anti-bot rate limiting, hammering it with 10 concurrent requests guarantees throttling
- Per-uni concurrency override: CSU=2, others=10
- Slower but more likely to succeed

**My recommendation: Option A.** API endpoint inspection is a 10-minute task and would make CSU scrape complete in <30 seconds for all 200 courses. The other options are workarounds for symptoms.
