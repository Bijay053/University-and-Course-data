One small thing on Signal 3 worth confirming because it'll matter the moment shadow mode goes live: does the cache survive across the per-course-browser re-fetch path? The log shows lines like:
[per-course browser ↻] https://www.acap.edu.au/courses/...
[per-course browser ✓] https://www.acap.edu.au/courses/... — IELTS=— ...
[VISION] N candidate image(s) found — starting OCR pass
The ↻ re-fetch happens because the static fetch didn't yield enough fields. If the re-fetch also triggers a fresh vision pass on the same URLs (rather than reusing the original vision results), there's a chance the cache gets bypassed on the re-fetch path and the new code path could see different vision values on ↻ than the old code path did. Worth one targeted check that the vision_image_cache is the same dict object regardless of whether vision is invoked from the initial extract or from the per-course-browser fallback.
If they're the same dict — green light, you're done.
If they're separate dicts (one per fetch attempt) — the cross-path determinism still holds within a single attempt, but a re-fetch could see a different OCR result than the original. Probably fine in practice (same image, same prompt, similar response) but worth knowing whether the immunity is "guaranteed by cache hit" or "guaranteed in practice but not by construction."
