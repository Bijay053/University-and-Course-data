## Why it's firing too late

Looking at the worker log flow on KBS:

1. Browser fetch (often times out at 20s)
2. Vision OCR runs on screenshots
3. Central page cache hit → fee + English populated from cached data
4. AI fallback runs for any STILL-missing fields
5. (Gemini-primary apparently sits somewhere in here but rarely has anything to do)

By the time the Gemini-primary check happens, the payload usually already has fee_schedule + english_requirements from cached central pages, fee from PDF, English from vision OCR. So Gemini sees "no missing fields, skip."

This is **late fallback behavior**, not "primary" behavior.

## What the original ticket asked for

From Ticket 2:
> "Hard fields — Gemini Flash as PRIMARY (not fallback): international_fee, domestic_fee, fee_term, duration_value/unit/text, ielts_overall, pte_overall, toefl_overall, cambridge_overall, duolingo_overall, sub_category, category, mode, intake_text"
>
> "Replace AI-fallback for fee/duration/IELTS/sub_category/mode with Gemini Flash as primary"

The word "PRIMARY" was the whole point. PR 2 wired Gemini as another late fallback, which means:
- It rarely fires (1/29 courses on KBS)
- It contributes nothing when it does fire (Study Abroad was rejected)
- The cost target was met but only because Gemini barely runs

## What I want from this fix

**Move the Gemini call earlier in the per-course pipeline:**

1. After course HTML is fetched (browser or static)
2. **BEFORE** the existing regex/CSS extractors run on hard fields
3. **BEFORE** vision OCR
4. **BEFORE** central page lookup pre-seeds the payload

Flow should be:
1. Fetch course HTML
2. Call `gemini_primary.extract_primary(html, url)` for hard fields
3. Pre-seed payload with Gemini's results
4. Run regex/CSS only for fields Gemini left null (true fallback)
5. Vision OCR runs only if both Gemini AND regex left fields null
6. Central pages still pre-seed at the start (they're per-uni, not per-course)

This way `extraction_method` should show `gemini_primary` for fees/IELTS/duration on most courses, with `regex`/`vision` only as fallback.

## Why this matters

The original architectural promise was "hard fields handled by Gemini, easy fields by regex." That's what makes the system robust to per-uni HTML differences — the same Gemini prompt works across ASA, KBS, USQ, CSU, UTAS, VIT, Torrens because it reads natural language. Regex/CSS rules break per-uni and require ongoing maintenance.

With current PR 2 wiring, we still rely on per-uni regex/CSS/vision rules for hard fields. The same brittle pattern that caused 9 commits this weekend will keep biting us when we expand to remaining unis.
