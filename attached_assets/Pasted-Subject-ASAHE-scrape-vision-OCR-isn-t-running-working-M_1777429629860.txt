So ielts_overall=6.5 for those Master courses comes from a different code path than the sibling cache build line. Possibilities:

A different sibling cache mechanism we don't see logged

The Evidence Review panel for Master AI shows ielts_overall=6.0 from uni_pdf:requirements. But the staging table shows 6.5. The staging column and the evidence row do not match for the same course.
This means the architecture's enforce_source_evidence guarantee is broken: there's a value in scraped_courses.ielts_overall (6.5) that has no selected=true evidence row backing it. Admins reviewing this course see a value they cannot trace.
Diagnostic SQL to identify which code path is writing 6.5:
sql-- Show every scraped_courses row + every evidence row for one specific course
WITH target AS (
  SELECT id, course_name, ielts_overall, pte_overall, toefl_overall,
         created_at, scrape_job_id
  FROM scraped_courses
  WHERE course_name ILIKE '%Master of Information Technology (Artificial Intelligence)%'
  ORDER BY created_at DESC
  LIMIT 5
)
SELECT 'COURSE' AS type,
       t.id::text AS row_id,
       t.scrape_job_id AS job,
       'ielts=' || COALESCE(t.ielts_overall::text,'NULL') ||
       ' pte=' || COALESCE(t.pte_overall::text,'NULL') ||
       ' toefl=' || COALESCE(t.toefl_overall::text,'NULL') AS info,
       t.created_at::text AS created
FROM target t
UNION ALL
SELECT 'EVIDENCE',
       sfe.id::text,
       sfe.method,
       sfe.field_key || '=' || COALESCE(sfe.normalized_value,'NULL') ||
       ' selected=' || sfe.selected::text,
       sfe.created_at::text
FROM scraped_field_evidence sfe
WHERE sfe.scraped_course_id IN (SELECT id FROM target)
  AND sfe.field_key IN ('ielts_overall','pte_overall','toefl_overall')
ORDER BY type, created DESC;
This shows whether multiple scraped_courses rows exist for the course, what each row's columns contain, and what evidence rows exist. The expected good state is: ONE course row, with column values matching ONE selected=true evidence row per field.
