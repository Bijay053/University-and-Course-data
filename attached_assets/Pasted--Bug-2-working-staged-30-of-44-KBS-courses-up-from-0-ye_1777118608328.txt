## Bug 1 (P0): UI URL overrides in Advanced section being ignored

**Repro:** In the Scraping page UI I clicked "Show advanced options" and filled in three fields:

- Fee Schedule Page URL: `https://www.kbs.edu.au/admissions/fees/international-fees`
- English Requirements Page URL: `https://www.kbs.edu.au/admissions/entry-requirements/english-entry-requirements`
- Academic Requirements Page URL: `https://www.kbs.edu.au/admissions/entry-requirements/international-entry-requirements`

Then clicked Start AI Scraping on KBS.

The scraper used the URL from `scrape_config.uniPages.feePage` in the database (set via SQL earlier) instead of the URLs I just typed. The Advanced fields had no effect on the scrape.

**Why this matters:** the whole point of the Advanced section is letting the user override DB defaults on a per-scrape basis. Right now those fields are decorative — typing in them does nothing. For unis where the configured fee page is wrong, or the user wants to test a different page without permanent DB changes, there's no way to do it. Verification of new unis becomes painful: instead of "type the URL and re-run," the workflow is "manually update DB row with SQL, restart, re-run."

**Diagnostic needed:** is this frontend (form values not being sent in `POST /api/scrape/start` request body) or backend (sent but ignored by orchestrator)? Quick check: open browser dev tools, Network tab, fire a scrape with Advanced filled in, look at the request payload. If `feePage` etc. aren't in the JSON → frontend. If they are → backend precedence issue.

**Precedence I want once fixed:**
1. Request-body overrides (UI Advanced) — highest, transient, one-scrape-only, NOT persisted to DB
2. `university.scrape_config.uniPages.*` — medium, persisted across runs
3. Auto-discovery (Track B) — fallback
4. None — fall through to staging gate behavior

## Bug 2 (P0): Duplicate courses staged — 9 dupes out of 30 staged

Staged list contains the same course twice:

```
Master of Accounting                              — twice
Master of Business Analytics                      — twice
Master of Business Analytics (Extension)          — twice
Master of Business Analytics (Information Tech)   — twice
Master of Information Technology                  — twice
Master of Information Technology (Extension)      — twice
Graduate Diploma of Business Analytics            — twice
Graduate Diploma of Business Administration       — twice
Bachelor of Information Technology                — twice
```

Root cause from the worker log: KBS exposes the same course via two different URL patterns:
- `https://www.kbs.edu.au/node/4129`  (Drupal node id)
- `https://www.kbs.edu.au/courses/master-of-information-technology`  (slug)

Discovery picks up BOTH, extraction runs on BOTH, both get staged as separate rows. Dedup is failing.

**Fix:** dedup by canonical URL (follow redirects + normalize) or by extracted course name + uni_id. The 30 "staged" is really 21 unique courses + 9 duplicates. Inflates the success metric and pollutes the staging table.

You'll see this on every Drupal-based uni site (and there are several in the AU sector).

KBS legitimately offers all of these as real MBA specialisations. They're NOT category landing pages — each one is a distinct program with its own admission, fees, duration. **Fix:** tighten the category_landing_page detector. Probably needs to look at whether the page has program-specific data (CRICOS code, intake dates, duration, fees) before classifying as landing page. If it has those → it's a real program even if URL/structure looks landing-y.

## Bug 4 (P0): Fee parser uses wrong column — Subject fee instead of Course fee

The KBS fee table has TWO columns:
- "Subject fee" (per-subject cost, e.g., $4,910 for MBA)
- "Course fee" (total program cost, e.g., $58,920 for MBA)

The parser is grabbing "Subject fee" and storing $4,910 as `international_fee` for the entire program. Result: `intl=4910.0 per=None` — the per field is NULL when it should be `subject` or `unit`.

This means every fee in the staged courses is **wrong by ~12x** (a typical KBS MBA is 12 subjects = $58,920, not $4,910).

**Fix options (pick one or do both):**
- (a) Prefer the `Course fee` column when present — it's the actual program total, which is what users want to compare
- (b) Record the per-unit context properly: when grabbing Subject fee, set `per=subject` and store subject count separately so total can be computed

Ideal: do (a) AND (b). Show "$58,920 total" in the UI but expose subject breakdown if available.

## Bug 5 (P1): Bachelor of Business rejected twice as online_only — false positive

```
Bachelor of Business: rejected: online_only  (twice — dedup bug strikes again)
```

KBS Bachelor of Business is a real on-campus program offered in Sydney, Melbourne, Brisbane, and Adelaide. The online_only filter is incorrectly classifying it.

Probably the Mode field defaulted to Online or got set to Online by AI fallback when the per-course page didn't have explicit on-campus signals. Then the online_only filter kicked it out.

This is similar in spirit to the KBS Bug 1 (mode default) ticket from earlier today — mode should be derived from location, not defaulted to Online.

**Fix:** when location_text contains a real city (Sydney, Melbourne, Brisbane, Adelaide), mode should be derived as on-campus or blended, not Online. Ideally check this BEFORE the online_only filter rejects.

We're calling AI to fill duration on every course (`AI filled 2 field(s): duration_value, duration_unit`) when the fee page already encodes it. Saves an AI call per course, gives us cleaner data.

Not urgent. Just noting because we're already parsing that table.
