The Master AI course displays IELTS=6.5, PTE=58, TOEFL=85 in the staging table. The evidence panel for the same course shows ielts_overall=6.0 from uni_pdf:requirements.
These don't match. The displayed value (6.5) is correct per the source image. The evidence (6.0) is stale or misattributed. The user can't tell what's actually in the database or where it came from.
This is what the architecture's enforce_source_evidence is supposed to prevent. Either:

The sibling cache is writing to scraped_courses columns without creating evidence rows, OR
Stale evidence from a previous scrape isn't being purged when new values are written

Please verify which one. The fix differs:

If sibling cache bypasses evidence: every cached value needs an evidence row with method sibling_cache:postgraduate (or similar) and source_url pointing at the original course's vision evidence URL
If old evidence isn't purged: when new values are written for a (course, field), old evidence rows for that field need to be marked superseded or deleted

Issue 3: Bachelor courses show phantom DET=100 from PDF
The actual Bachelor image (linked below) lists IELTS, PTE, TOEFL, CAE, Cambridge Proficiency. NO Duolingo. NO DET row.
But every Bachelor in the staging table shows DET=100, sourced from uni_pdf:requirements. The PDF parser is including a Duolingo row that doesn't apply to these courses — probably a generic Duolingo policy entry from a different part of the PDF being mis-mapped.
Two-step fix:

Once vision OCR runs on the Bachelor image, the Bachelor image becomes the authoritative tier-4 source. With no DET row in the image, DET stays null. PDF tier-2 cannot override.
But until vision OCR fires on Bachelor pages, PDF still wins. We need vision running on Bachelor pages too — likely the same root cause as Issue 2 (vision isn't running consistently).
