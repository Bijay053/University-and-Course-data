**On the bigger picture:**

The last few exchanges have featured "three fixes shipped, paste-and-run script, half-applied bundle". That's a pattern. The architecture work over the past two weeks built three things to prevent exactly this: per-uni configs, the regression sweep, the expected-diff workflow. They only protect us if we use them.

Option C through Phase 5 each followed the discipline: one change, one PR, one sweep, one merge gate. The result was confidence that no other uni broke. The recent commits skipped that and produced the half-applied state we're now untangling.

Going forward: **one migration per PR, one sweep per migration, one merge gate per sweep.** No bundling of "a few small fixes together because they're related." If two changes are related, they sweep together but with an enumerated expected-dirty list that covers both. If they're unrelated, they ship separately.

The rollback-then-sequenced-redeploy I outlined above is the correction for the current state. After that's done, the queue continues at low cost per migration because each one is small and isolated.
